📖 Introducción Conceptual

iptables es el cortafuegos estándar del kernel Linux, activo desde la versión 2.4. Actúa como un filtro de paquetes a nivel de kernel: intercepta cada paquete de red y aplica reglas para decidir qué hacer con él (permitirlo, bloquearlo, redirigirlo, registrarlo…).

💡

Desde Linux 5.x convive con nftables (su sucesor moderno), pero iptables sigue siendo ampliamente utilizado en producción y es requerido en certificaciones como LPIC-2 y RHCE.

🗂️ Las cuatro tablas

iptables organiza las reglas en tablas, cada una con un propósito bien definido:

🔍
filter
Tabla por defecto
Decide qué paquetes se permiten o bloquean. La más utilizada en la práctica.
🔄
nat
Traducción de direcciones
NAT, DNAT, SNAT y reenvío de puertos. Modifica IP/puerto de origen o destino.
🔧
mangle
Modificación de cabeceras
Altera campos del paquete: TTL, TOS, marcas para QoS. Uso avanzado.
⚡
raw
Exención de conntrack
Excluye paquetes del seguimiento de conexiones para mayor rendimiento.

⛓️ Las cinco cadenas predefinidas

Cada tabla tiene cadenas que se activan según el momento en que el paquete recorre el kernel:

📥 INPUT → paquetes para el propio equipo
📤 OUTPUT → paquetes generados localmente
🔀 FORWARD → paquetes en tránsito (router)
🔺 PREROUTING → antes de decidir el destino
🔻 POSTROUTING → justo antes de salir

📦 Paquete entrante desde la red
🌐 RedEntrada
→

Regla de oro: las reglas se evalúan en orden, de arriba hacia abajo. En cuanto una regla coincide, se aplica su acción y se detiene la evaluación (excepto LOG, que continúa al siguiente).

⚙️ Instalación y Verificación

🔍 Verificar si está instalado

bash
# Comprobar versión instalada
iptables --version

# Listar reglas activas con contadores y números de línea
iptables -L -v -n --line-numbers

🐧 Instalación por distribución

🟠 Debian / Ubuntu
sudo apt update
sudo apt install \
  iptables \
  iptables-persistent
🔵 RHEL / CentOS / Fedora
# Detener firewalld primero
sudo systemctl stop firewalld
sudo systemctl disable firewalld
sudo dnf install iptables-services
sudo systemctl enable --now iptables

📋 Ver reglas y políticas actuales

bash
# Tabla filter con contadores y números de línea
iptables -t filter -L -v -n --line-numbers

# Tabla NAT
iptables -t nat -L -v -n

# Ver TODAS las tablas
for t in filter nat mangle raw; do
  echo "=== Tabla: $t ==="
  iptables -t $t -L -n
done
⚠️

La política por defecto (ACCEPT o DROP) se aplica cuando ninguna regla coincide. Ejecutar iptables -P INPUT DROP sin reglas de whitelist previas puede dejarte sin acceso al servidor.

✏️ Sintaxis Básica

🏗️ Anatomía de un comando

iptables [-t tabla] COMANDO [cadena] [condiciones] -j ACCIÓN
Comando base
Tabla (opcional, default: filter)
Subcomando (-A, -I, -D…)
Cadena (INPUT, OUTPUT…)
Condiciones de coincidencia
Acción (ACCEPT, DROP…)

📋 Comandos principales

ComandoDescripciónEjemplo
-A cadena➕ Append — añade al final-A INPUT -j ACCEPT
-I cadena [N]⬆️ Insert — inserta en posición N-I INPUT 1 -j DROP
-D cadena N🗑️ Delete — elimina la regla N-D INPUT 3
-R cadena N♻️ Replace — reemplaza la regla N-R INPUT 2 ...
-L [cadena]📋 List — lista las reglas-L INPUT -n -v
-F [cadena]🧹 Flush — borra todas las reglas-F INPUT
-X [cadena]💥 eXpunge — elimina cadenas de usuario-X MI_CADENA
-P cadena acción🎯 Policy — política por defecto-P INPUT DROP
-N cadena🆕 New — crea cadena personalizada-N SSH_GUARD
-Z [cadena]🔢 Zero — resetea contadores-Z INPUT

🎯 Condiciones (matches) comunes

OpciónDescripción
-s IP/máscara🌐 IP o red de origen
-d IP/máscara🌐 IP o red de destino
-p protocolo📡 Protocolo: tcp, udp, icmp, all
--dport puerto🔌 Puerto de destino (requiere -p tcp/udp)
--sport puerto🔌 Puerto de origen
-i interfaz🖧 Interfaz de entrada (eth0, lo…)
-o interfaz🖧 Interfaz de salida
-m state --state🔗 Estado conntrack: NEW, ESTABLISHED, RELATED
-m multiport --dports🎛️ Múltiples puertos: 80,443,8080
-m limit --limit⏱️ Límite de tasa (rate limiting)
-m recent📊 Seguimiento de IPs recientes (anti brute-force)
! condición🚫 Negación: todo EXCEPTO lo indicado

🔧 Ejemplos Prácticos

①
🔒 Bloquear todo excepto SSH
filter · INPUT · política DROP

Configuración básica de servidor: permite solo el puerto 22 y bloquea todo lo demás. El orden importa: SSH va antes del DROP.

bash
# 1. CRÍTICO: permitir SSH ANTES de cambiar política a DROP
iptables -A INPUT -p tcp --dport 22 -m state --state NEW,ESTABLISHED -j ACCEPT

# 2. Permitir respuestas de conexiones ya establecidas
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 3. Ahora es seguro poner la política en DROP
iptables -P INPUT   DROP
iptables -P FORWARD DROP
②
🔁 Permitir tráfico loopback
filter · INPUT/OUTPUT · interfaz lo

El tráfico por la interfaz lo (127.0.0.1) siempre debe estar permitido. Muchos servicios —bases de datos, APIs— se comunican internamente por ella.

bash
# Permitir todo tráfico en interfaz loopback
iptables -A INPUT  -i lo -j ACCEPT
iptables -A OUTPUT -o lo -j ACCEPT

# Anti-spoofing: bloquear paquetes de otra interfaz que finjan ser 127.x
iptables -A INPUT ! -i lo -s 127.0.0.0/8 -j DROP
③
⏱️ Rate limiting SSH (anti-fuerza bruta)
filter · módulo recent · cadena personalizada

Limita nuevas conexiones SSH a 4 por minuto por IP. Las que superen el límite son bloqueadas y registradas en syslog.

bash
# Cadena personalizada para control de acceso SSH
iptables -N SSH_CONTROL

# Si la IP supera 4 intentos en 60s → LOG + DROP
iptables -A SSH_CONTROL -m recent --name SSH --rcheck \
         --seconds 60 --hitcount 4 \
         -j LOG --log-prefix "SSH_BRUTE: "
iptables -A SSH_CONTROL -m recent --name SSH --rcheck \
         --seconds 60 --hitcount 4 -j DROP

# Registrar IP y aceptar (dentro del límite)
iptables -A SSH_CONTROL -m recent --name SSH --set -j ACCEPT

# Redirigir nuevas conexiones SSH a la cadena de control
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -j SSH_CONTROL
④
🔌 Redirigir puerto 8080 → 80 (DNAT)
nat · PREROUTING · DNAT

Útil cuando el servidor web corre en el puerto 80 pero quieres exponer el 8080 externamente sin modificar la aplicación.

bash
# Redirigir en PREROUTING (antes del routing)
iptables -t nat -A PREROUTING \
         -p tcp --dport 8080 \
         -j DNAT --to-destination :80

# Permitir el tráfico en el puerto destino real
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Para que también funcione en el mismo equipo (loopback)
iptables -t nat -A OUTPUT \
         -p tcp --dport 8080 \
         -j DNAT --to-destination 127.0.0.1:80
⑤
🌐 Compartir Internet con NAT (MASQUERADE)
nat · POSTROUTING · ip_forward

Convierte el equipo en un gateway para que una red interna acceda a Internet usando la IP pública dinámica del gateway.

bash
# 1. Activar reenvío de paquetes entre interfaces (obligatorio)
echo 1 > /proc/sys/net/ipv4/ip_forward
# Persistente: agregar "net.ipv4.ip_forward=1" en /etc/sysctl.conf

# 2. MASQUERADE: enmascara IPs internas con la IP pública de eth0
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

# 3. Reenvío LAN → Internet
iptables -A FORWARD -i eth1 -o eth0 -j ACCEPT

# 4. Tráfico de retorno (respuestas desde Internet)
iptables -A FORWARD -i eth0 -o eth1 \
         -m state --state ESTABLISHED,RELATED -j ACCEPT
⑥
🚫 Bloquear una IP específica
filter · INPUT · -I (insertar al inicio)

Bloquea inmediatamente una IP atacante. Usa -I para insertar en posición 1 con máxima prioridad.

bash
# Bloquear IP específica en posición 1 (máxima prioridad)
iptables -I INPUT 1 -s 203.0.113.45 -j DROP

# Bloquear un rango CIDR completo
iptables -I INPUT 1 -s 203.0.113.0/24 -j DROP

# Verificar que quedó en posición 1
iptables -L INPUT -n --line-numbers

# Desbloquear (eliminar por número de línea)
iptables -D INPUT 1

🧪 Ejercicios Resueltos

Lee el enunciado, intenta resolverlo por tu cuenta y luego despliega la solución haciendo clic.

🅐 Firewall web público + SSH restringido por subred▼
Configura un firewall que solo permita HTTP (80) y HTTPS (443) desde cualquier origen, y SSH (22) únicamente desde 192.168.1.0/24. Bloquear el resto.
✅ Solución
bash
#!/bin/bash
# ── Limpiar reglas previas ──
iptables -F
iptables -X

# ── Políticas por defecto restrictivas ──
iptables -P INPUT   DROP
iptables -P FORWARD DROP
iptables -P OUTPUT  ACCEPT

# ── Loopback siempre permitido ──
iptables -A INPUT -i lo -j ACCEPT

# ── Conexiones ya establecidas ──
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# ── SSH solo desde la subred interna ──
iptables -A INPUT -p tcp -s 192.168.1.0/24 --dport 22 -j ACCEPT

# ── Web desde cualquier origen ──
iptables -A INPUT -p tcp -m multiport --dports 80,443 -j ACCEPT

# ── Registrar lo que cae (opcional) ──
iptables -A INPUT -j LOG --log-prefix "FW_DROP: " --log-level 4
🅑 Prevenir fuerza bruta SSH (máx. 3 intentos/min)▼
Bloquea automáticamente cualquier IP que supere 3 intentos de conexión SSH en menos de 60 segundos.
✅ Solución
bash
# Cadena dedicada al anti-brute-force
iptables -N SSH_BF

# Si la IP tiene 3+ intentos en 60s → LOG y DROP
iptables -A SSH_BF -m recent --name SSHBF \
         --rcheck --seconds 60 --hitcount 3 \
         -j LOG --log-prefix "SSH_BF_BLOCK: "
iptables -A SSH_BF -m recent --name SSHBF \
         --rcheck --seconds 60 --hitcount 3 -j DROP

# Registrar IP y aceptar (dentro del límite)
iptables -A SSH_BF -m recent --name SSHBF --set -j ACCEPT

# Solo conexiones NUEVAS van al control
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -j SSH_BF
🅒 Router NAT con IP pública dinámica (gateway)▼
Configura el equipo como gateway NAT para que la red 10.0.0.0/24 (eth1) salga a Internet por eth0 con IP pública dinámica.
✅ Solución
bash
#!/bin/bash
# Activar IP forwarding
sysctl -w net.ipv4.ip_forward=1
echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf

iptables -F
iptables -t nat -F

# MASQUERADE (IP dinámica → usa la IP actual de eth0)
iptables -t nat -A POSTROUTING \
         -s 10.0.0.0/24 -o eth0 -j MASQUERADE

# Reenvío LAN → Internet
iptables -A FORWARD \
         -i eth1 -o eth0 -s 10.0.0.0/24 -j ACCEPT

# Reenvío: respuestas de Internet → LAN
iptables -A FORWARD \
         -i eth0 -o eth1 \
         -m state --state ESTABLISHED,RELATED -j ACCEPT
🅓 Port forwarding: puerto 2222 → servidor interno 10.0.0.5:22▼
Redirige el tráfico TCP del puerto 2222 del gateway hacia el puerto 22 del servidor interno 10.0.0.5.
✅ Solución
bash
# DNAT: redirigir :2222 → 10.0.0.5:22
iptables -t nat -A PREROUTING \
         -p tcp --dport 2222 \
         -j DNAT --to-destination 10.0.0.5:22

# Permitir reenvío hacia el servidor interno
iptables -A FORWARD \
         -p tcp -d 10.0.0.5 --dport 22 \
         -m state --state NEW,ESTABLISHED,RELATED \
         -j ACCEPT

sysctl -w net.ipv4.ip_forward=1
🅔 Registrar paquetes ICMP entrantes y bloquearlos▼
Registra todos los paquetes ICMP (ping) entrantes con prefijo ICMP_IN: y luego bloquéalos sin responder al emisor.
✅ Solución
bash
# LOG primero: no detiene la evaluación, el paquete continúa
iptables -A INPUT \
         -p icmp \
         -j LOG \
         --log-prefix "ICMP_IN: " \
         --log-level 4

# DROP silencioso: el emisor no recibe respuesta (el equipo "no existe")
iptables -A INPUT -p icmp -j DROP

# Ver logs en tiempo real:
# journalctl -f | grep ICMP_IN
🅕 Reglas persistentes con iptables-save / restore▼
Guarda las reglas actuales para que sobrevivan a un reinicio, tanto en Debian/Ubuntu como en RHEL/CentOS.
✅ Solución
bash
# ══ OPCIÓN A: Manual (cualquier distro) ══
iptables-save  > /etc/iptables/rules.v4
iptables-restore  < /etc/iptables/rules.v4

# ══ OPCIÓN B: iptables-persistent (Debian/Ubuntu) ══
apt install iptables-persistent
netfilter-persistent save
systemctl enable netfilter-persistent

# ══ OPCIÓN C: Servicio systemd personalizado ══
# /etc/systemd/system/iptables-restore.service
[Unit]
Description=Restaurar reglas iptables
Before=network-pre.target

[Service]
Type=oneshot
ExecStart=/sbin/iptables-restore /etc/iptables/rules.v4
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target

systemctl enable --now iptables-restore

💾 Persistencia de Reglas

⚠️

Las reglas de iptables son volátiles por defecto: desaparecen al reiniciar. Es imprescindible usar alguno de los métodos siguientes en producción.

💾 iptables-save y iptables-restore

bash
# Exportar todas las tablas a archivo
iptables-save > /etc/iptables/rules.v4

# Restaurar desde archivo
iptables-restore < /etc/iptables/rules.v4

# Ver contenido del archivo guardado
cat /etc/iptables/rules.v4

📦 netfilter-persistent (Debian/Ubuntu)

bash
apt install iptables-persistent

# Guardar reglas actuales en /etc/iptables/rules.v4
netfilter-persistent save

# Recargar reglas guardadas
netfilter-persistent reload

# Estado del servicio
systemctl status netfilter-persistent

🔧 Script en /etc/network/if-pre-up.d/

bash
# Se ejecuta antes de levantar interfaces de red
cat > /etc/network/if-pre-up.d/iptables <<'EOF'
#!/bin/bash
iptables-restore < /etc/iptables/rules.v4
EOF
chmod +x /etc/network/if-pre-up.d/iptables
ℹ️

En RHEL/CentOS con iptables-services, las reglas se guardan en /etc/sysconfig/iptables. Usa service iptables save y systemctl enable iptables.

✅ Consejos de Seguridad y Buenas Prácticas

🔑
Whitelist SSH ANTES de cambiar la política a DROP

Añade la regla de aceptación SSH antes de iptables -P INPUT DROP. De lo contrario perderás acceso al servidor remoto permanentemente hasta tener consola física.

📊
Orden: reglas específicas primero, generales después

Las reglas concretas (por IP, por puerto) deben preceder a las genéricas. El motor evalúa en orden y se detiene en la primera coincidencia.

🧹
Vacía las reglas antes de aplicar un script nuevo

Inicia con iptables -F y iptables -X para evitar reglas duplicadas o conflictivas de ejecuciones anteriores.

⚡
Usa -n en los listados para mayor velocidad

iptables -L -n -v evita la resolución DNS inversa, lo que puede ser muy lento en servidores con DNS mal configurado.

🔗
Aprovecha el seguimiento de estado (conntrack)

-m state --state ESTABLISHED,RELATED -j ACCEPT es fundamental: permite el tráfico de retorno sin reglas bidireccionales para cada servicio.

⏰
Usa un cron de seguridad al probar reglas en remoto

Antes de aplicar reglas arriesgadas: echo "iptables -F && iptables -P INPUT ACCEPT" | at now + 5 min. Cancela el job si todo funciona.

⬆️
Usa -I para bloqueos urgentes

-I INPUT 1 inserta al inicio (máxima prioridad). Ideal para bloqueos de emergencia. -A añade al final del flujo normal de reglas.

📝
Documenta tus reglas con prefijos de LOG descriptivos

Comenta cada bloque en tus scripts y usa --log-prefix "NOMBRE_REGLA: " para identificar fácilmente el origen en los logs del sistema.

📄 Plantilla de script de firewall recomendada

bash
#!/bin/bash
# firewall.sh — Plantilla segura y comentada
set -e  # Detener si hay error

# ══ 1. Limpiar reglas existentes ══
iptables -F
iptables -X
iptables -t nat -F

# ══ 2. Políticas permisivas al inicio (seguridad) ══
iptables -P INPUT   ACCEPT
iptables -P FORWARD ACCEPT
iptables -P OUTPUT  ACCEPT

# ══ 3. Reglas específicas ══
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -p tcp --dport 22  -j ACCEPT  # SSH
iptables -A INPUT -p tcp --dport 80  -j ACCEPT  # HTTP
iptables -A INPUT -p tcp --dport 443 -j ACCEPT  # HTTPS

# ══ 4. Políticas restrictivas AL FINAL ══
iptables -P INPUT   DROP
iptables -P FORWARD DROP

# ══ 5. Guardar para persistencia ══
iptables-save > /etc/iptables/rules.v4
echo "✅ Firewall aplicado correctamente."

🚨 Posibles Problemas y Soluciones

🔒 Me quedé sin acceso SSH al servidor▼
✅ Solución

Desde consola física, VNC/KVM o panel de emergencia del proveedor cloud:

bash
# Limpiar reglas y restaurar acceso inmediatamente
iptables -F
iptables -P INPUT ACCEPT

En cloud: EC2 Serial Console, Azure Bastion, Google Cloud SSH in Browser, consola web del VPS.

🔄 Las reglas desaparecen tras reiniciar▼
✅ Solución

Las reglas son volátiles por diseño. Usa iptables-persistent en Debian/Ubuntu, service iptables save en RHEL, o un servicio systemd personalizado (ver sección 06).

🌐 El NAT funciona pero los equipos internos no salen a Internet▼
✅ Solución
bash
# 1. Verificar ip_forward (debe devolver 1)
cat /proc/sys/net/ipv4/ip_forward

# 2. Activarlo si es 0
echo 1 > /proc/sys/net/ipv4/ip_forward

# 3. Verificar cadena FORWARD
iptables -L FORWARD -n -v

# 4. Verificar regla MASQUERADE
iptables -t nat -L POSTROUTING -v -n
🔌 El DNAT no funciona desde el propio gateway▼
✅ Solución

PREROUTING no aplica al tráfico originado localmente. Añade una regla en OUTPUT de la tabla nat:

bash
iptables -t nat -A OUTPUT \
         -p tcp --dport 2222 \
         -j DNAT --to-destination 10.0.0.5:22
💥 Error: "No chain/target/match by that name"▼
✅ Solución

El módulo del kernel requerido no está cargado. Cárgalo manualmente:

bash
modprobe xt_recent    # módulo 'recent' (anti brute-force)
modprobe ip_conntrack  # módulo conntrack / state
lsmod | grep xt_      # ver módulos netfilter cargados
⚡ Conflicto con firewalld o ufw▼
✅ Solución

iptables, firewalld y ufw gestionan netfilter simultáneamente y se interfieren. Elige uno y desactiva los demás:

bash
# Deshabilitar firewalld (RHEL/CentOS)
systemctl stop firewalld
systemctl mask firewalld

# Deshabilitar ufw (Debian/Ubuntu)
ufw disable
systemctl disable ufw
🧪

Entornos de prueba: practica siempre en máquinas virtuales (VirtualBox, KVM, VMware) o contenedores Docker/LXC antes de aplicar estas configuraciones en producción. Un error en un servidor remoto real puede dejarlo inaccesible.
