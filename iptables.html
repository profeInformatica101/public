01 Introducción Conceptual

iptables es el cortafuegos estándar del kernel Linux, integrado desde la versión 2.4. Actúa como un filtro de paquetes a nivel de kernel: intercepta cada paquete de red y aplica reglas para decidir qué hacer con él (permitirlo, bloquearlo, redirigirlo, etc.).

Desde Linux 5.x convive con nftables (su sucesor), pero iptables sigue siendo el más usado en producción y en certificaciones como LPIC o RHCE.

Las cuatro tablas

iptables organiza las reglas en tablas, cada una con un propósito específico:

TablaPropósito principalCadenas disponibles
filterFiltrado de paquetes (tabla por defecto)INPUT, OUTPUT, FORWARD
natTraducción de direcciones (NAT, reenvío de puertos)PREROUTING, OUTPUT, POSTROUTING
mangleModificar cabeceras de paquetes (QoS, TTL…)Todas las cadenas
rawExención de seguimiento de conexiones (conntrack)PREROUTING, OUTPUT

Las cinco cadenas predefinidas

INPUT OUTPUT FORWARD PREROUTING POSTROUTING
Cadena¿Cuándo actúa?
INPUTPaquetes destinados al propio equipo
OUTPUTPaquetes originados en el propio equipo
FORWARDPaquetes en tránsito (router/gateway)
PREROUTINGAntes de decidir el destino (útil para DNAT)
POSTROUTINGJusto antes de salir por la interfaz (útil para SNAT/MASQUERADE)

💡 Regla de oro: las reglas se evalúan en orden. En cuanto una coincide, se aplica su acción y se deja de evaluar el resto (salvo LOG, que continúa).

02 Instalación y Verificación

Verificar si está instalado

# Comprobar versión instalada
iptables --version

# Listar reglas activas (tabla filter)
iptables -L -v -n

Instalación en Debian/Ubuntu

sudo apt update
sudo apt install iptables iptables-persistent

Instalación en RHEL/CentOS/Fedora

# En sistemas con firewalld, primero detenerlo
sudo systemctl stop firewalld
sudo systemctl disable firewalld

sudo dnf install iptables-services
sudo systemctl enable --now iptables

Ver reglas actuales y políticas por defecto

# Ver tabla filter con nombres (no IPs resueltas) y contadores
iptables -t filter -L -v -n --line-numbers

# Ver tabla NAT
iptables -t nat -L -v -n
⚠️ La política por defecto (ACCEPT o DROP) se aplica cuando ninguna regla coincide. Cámbiala con cuidado: iptables -P INPUT DROP puede bloquearte de inmediato si no tienes reglas de whitelist.

03 Sintaxis Básica

Estructura general

iptables [-t tabla] COMANDO [cadena] [condiciones] -j ACCIÓN

Si no se especifica -t tabla, se usa filter por defecto.

Comandos principales

ComandoDescripción
-A cadenaAppend — añade regla al final de la cadena
-I cadena [N]Insert — inserta en la posición N (1 = inicio)
-D cadena NDelete — elimina la regla número N
-R cadena NReplace — reemplaza la regla número N
-L [cadena]List — lista las reglas
-F [cadena]Flush — borra todas las reglas de la cadena
-X [cadena]eXpunge — elimina cadenas de usuario
-P cadena acciónPolicy — establece política por defecto
-Z [cadena]Zero — resetea contadores de paquetes/bytes
-N cadenaNew — crea una cadena personalizada

Condiciones (matches) comunes

OpciónDescripción
-s IP/máscaraIP de origen
-d IP/máscaraIP de destino
-p protocoloProtocolo: tcp, udp, icmp, all
--dport puertoPuerto de destino (requiere -p tcp/udp)
--sport puertoPuerto de origen
-i interfazInterfaz de entrada (eth0, lo…)
-o interfazInterfaz de salida
-m state --stateEstado de conexión (NEW, ESTABLISHED, RELATED)
-m multiport --dportsMúltiples puertos de destino
-m limit --limitLímite de tasa (rate limiting)
! condiciónNegación de la condición

Acciones (-j) principales

ACCEPT DROP REJECT LOG MASQUERADE DNAT SNAT RETURN
AcciónEfecto
ACCEPTPermite el paquete
DROPDescarta silenciosamente (sin respuesta al emisor)
REJECTDescarta y envía ICMP de error al emisor
LOGRegistra en syslog y continúa evaluando reglas
MASQUERADENAT dinámico (IP pública variable); solo en POSTROUTING
SNAT --to-sourceNAT con IP de origen fija
DNAT --to-destinationCambia IP/puerto de destino; solo en PREROUTING
RETURNVuelve a la cadena llamante

04 Ejemplos Prácticos

① Bloquear todo excepto SSH

Configuración básica de un servidor: permitir solo el puerto 22 en entrada y bloquear el resto.

bash# 1. Primero permitir SSH para no bloquearnos
iptables -A INPUT -p tcp --dport 22 -m state --state NEW,ESTABLISHED -j ACCEPT

# 2. Permitir tráfico de salida relacionado con SSH
iptables -A OUTPUT -p tcp --sport 22 -m state --state ESTABLISHED -j ACCEPT

# 3. Permitir tráfico de retorno ya establecido
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 4. Ahora sí, cambiar política por defecto a DROP
iptables -P INPUT DROP
iptables -P FORWARD DROP

② Permitir tráfico loopback

El tráfico por la interfaz lo (localhost) siempre debe estar permitido; muchos servicios se comunican internamente por ella.

bash# Permitir todo tráfico en interfaz loopback
iptables -A INPUT  -i lo -j ACCEPT
iptables -A OUTPUT -o lo -j ACCEPT

# También bloquear paquetes que lleguen por otra interfaz
# pero pretendan venir de 127.0.0.0/8 (anti-spoofing)
iptables -A INPUT ! -i lo -s 127.0.0.0/8 -j DROP

③ Rate limiting SSH (anti fuerza bruta)

Limitar nuevas conexiones SSH a 4 por minuto por IP de origen:

bash# Crear una cadena personalizada para el control SSH
iptables -N SSH_CONTROL

# Si supera 4 conexiones/min, descartar con LOG
iptables -A SSH_CONTROL -m recent --name SSH --rcheck \
         --seconds 60 --hitcount 4 -j LOG --log-prefix "SSH_BRUTE: "
iptables -A SSH_CONTROL -m recent --name SSH --rcheck \
         --seconds 60 --hitcount 4 -j DROP

# Registrar y permitir conexiones dentro del límite
iptables -A SSH_CONTROL -m recent --name SSH --set -j ACCEPT

# Redirigir nuevas conexiones SSH a la cadena de control
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -j SSH_CONTROL

④ Redirigir puerto 8080 → 80 (DNAT)

Muy útil cuando un servidor web corre en el puerto 80 pero queremos recibir en 8080:

bash# En la tabla nat, cadena PREROUTING
iptables -t nat -A PREROUTING \
         -p tcp --dport 8080 \
         -j DNAT --to-destination :80

# Permitir el tráfico redirigido en la cadena filter
iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Para redirigir también tráfico local (loopback)
iptables -t nat -A OUTPUT \
         -p tcp --dport 8080 \
         -j DNAT --to-destination 127.0.0.1:80

⑤ Compartir Internet con NAT (MASQUERADE)

Convertir el equipo en router/gateway para una red interna:

bash# 1. Activar reenvío de paquetes en el kernel
echo 1 > /proc/sys/net/ipv4/ip_forward
# Para hacerlo permanente, editar /etc/sysctl.conf:
# net.ipv4.ip_forward = 1

# 2. MASQUERADE en la interfaz de salida (ej: eth0 con IP pública)
iptables -t nat -A POSTROUTING \
         -o eth0 \
         -j MASQUERADE

# 3. Permitir reenvío desde la red interna (eth1) hacia Internet
iptables -A FORWARD -i eth1 -o eth0 -j ACCEPT

# 4. Permitir tráfico de retorno ya establecido
iptables -A FORWARD -i eth0 -o eth1 \
         -m state --state ESTABLISHED,RELATED -j ACCEPT

⑥ Bloquear una IP específica

bash# Bloquear todo tráfico entrante de una IP atacante
iptables -I INPUT 1 -s 203.0.113.45 -j DROP

# Bloquear un rango CIDR completo
iptables -I INPUT 1 -s 203.0.113.0/24 -j DROP

# Verificar que la regla quedó en posición 1
iptables -L INPUT -n --line-numbers

05 Ejercicios Resueltos

Lee el enunciado, intenta resolverlo y luego despliega la solución haciendo clic en ▶ Ver solución.

Ejercicio A — Firewall web + SSH restringido por subred
Configura un firewall que solo permita: tráfico HTTP (80), HTTPS (443) desde cualquier origen, y SSH (22) únicamente desde la subred 192.168.1.0/24. Todo lo demás debe bloquearse.
▸ Solución 
#!/bin/bash
# Firewall: web público + SSH solo desde LAN

# ── Limpiar reglas previas ──
iptables -F
iptables -X

# ── Políticas por defecto ──
iptables -P INPUT   DROP
iptables -P FORWARD DROP
iptables -P OUTPUT  ACCEPT

# ── Loopback ──
iptables -A INPUT -i lo -j ACCEPT

# ── Tráfico ya establecido ──
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# ── SSH solo desde 192.168.1.0/24 ──
iptables -A INPUT -p tcp -s 192.168.1.0/24 --dport 22 -j ACCEPT

# ── Web desde cualquier origen ──
iptables -A INPUT -p tcp -m multiport --dports 80,443 -j ACCEPT

# ── El resto cae en la política DROP (log opcional) ──
iptables -A INPUT -j LOG --log-prefix "FW_DROP: " --log-level 4
Ejercicio B — Prevenir fuerza bruta SSH (máx. 3 intentos/min)
Crea reglas que bloqueen automáticamente a cualquier IP que supere 3 intentos de conexión SSH nuevos en menos de 60 segundos.
▸ Solución 
#!/bin/bash
# Anti-brute-force SSH con módulo recent

# Cadena de control
iptables -N SSH_BF

# Si ya está en la lista de "atacantes", DROP
iptables -A SSH_BF -m recent --name SSHBF \
         --rcheck --seconds 60 --hitcount 3 \
         -j LOG --log-prefix "SSH_BF_BLOCKED: "
iptables -A SSH_BF -m recent --name SSHBF \
         --rcheck --seconds 60 --hitcount 3 -j DROP

# Registrar la IP y aceptar (dentro del límite)
iptables -A SSH_BF -m recent --name SSHBF --set -j ACCEPT

# Enviar nuevas conexiones SSH a la cadena SSH_BF
iptables -A INPUT -p tcp --dport 22 \
         -m state --state NEW -j SSH_BF
Ejercicio C — Router NAT con IP pública dinámica
Configura el equipo como gateway NAT para que los equipos de la red interna 10.0.0.0/24 (interfaz eth1) salgan a Internet por la interfaz eth0 con IP pública dinámica.
▸ Solución 
#!/bin/bash
# Gateway NAT con MASQUERADE

# ── Activar IP forwarding ──
sysctl -w net.ipv4.ip_forward=1
# Persistente: echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf

# ── Limpiar ──
iptables -F
iptables -t nat -F

# ── NAT de salida (MASQUERADE porque la IP pública es dinámica) ──
iptables -t nat -A POSTROUTING \
         -s 10.0.0.0/24 \
         -o eth0 \
         -j MASQUERADE

# ── Permitir reenvío LAN → Internet ──
iptables -A FORWARD \
         -i eth1 -o eth0 \
         -s 10.0.0.0/24 \
         -j ACCEPT

# ── Permitir tráfico de retorno ──
iptables -A FORWARD \
         -i eth0 -o eth1 \
         -m state --state ESTABLISHED,RELATED \
         -j ACCEPT
Ejercicio D — Port forwarding: puerto 2222 → servidor interno 10.0.0.5:22
Redirige el tráfico TCP que llegue al puerto 2222 del gateway hacia el puerto 22 del servidor interno 10.0.0.5.
▸ Solución 
#!/bin/bash
# Port forwarding: gateway:2222 → 10.0.0.5:22

# ── Redirigir en PREROUTING (tabla nat) ──
iptables -t nat -A PREROUTING \
         -p tcp \
         --dport 2222 \
         -j DNAT --to-destination 10.0.0.5:22

# ── Permitir el reenvío hacia el servidor interno ──
iptables -A FORWARD \
         -p tcp -d 10.0.0.5 --dport 22 \
         -m state --state NEW,ESTABLISHED,RELATED \
         -j ACCEPT

# ── Asegurarse de que IP forwarding está activo ──
sysctl -w net.ipv4.ip_forward=1
Ejercicio E — Registrar paquetes ICMP y bloquearlos
Registra todos los paquetes ICMP (ping) entrantes en syslog con el prefijo ICMP_IN: y luego bloquéalos sin responder al emisor.
▸ Solución 
#!/bin/bash
# LOG + DROP de ICMP entrante

# ── Registrar primero (LOG no detiene el procesamiento) ──
iptables -A INPUT \
         -p icmp \
         -j LOG \
         --log-prefix "ICMP_IN: " \
         --log-level 4

# ── Luego descartar (DROP = sin respuesta) ──
iptables -A INPUT -p icmp -j DROP

# Ver los logs en tiempo real:
# journalctl -f | grep ICMP_IN
# o: tail -f /var/log/kern.log | grep ICMP_IN
Ejercicio F — Reglas persistentes con iptables-save / restore
Guarda las reglas actuales de forma persistente para que sobrevivan a un reinicio del sistema, tanto en sistemas Debian/Ubuntu como RHEL/CentOS.
▸ Solución 
#!/bin/bash
# ══ OPCIÓN A: iptables-save manual ══

# Guardar reglas en archivo
iptables-save > /etc/iptables/rules.v4
ip6tables-save > /etc/iptables/rules.v6

# Restaurar manualmente
iptables-restore < /etc/iptables/rules.v4

# ══ OPCIÓN B: iptables-persistent (Debian/Ubuntu) ══

# Instalar
apt install iptables-persistent

# Guardar automáticamente (guarda las reglas actuales)
netfilter-persistent save

# El servicio las restaura en cada arranque
# Estado del servicio:
systemctl status netfilter-persistent

# ══ OPCIÓN C: Script systemd ══

# /etc/systemd/system/iptables-restore.service
[Unit]
Description=Restore iptables rules
Before=network-pre.target

[Service]
Type=oneshot
ExecStart=/sbin/iptables-restore /etc/iptables/rules.v4
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target

# Activar el servicio
systemctl enable --now iptables-restore

06 Persistencia de Reglas

Por defecto, las reglas de iptables son volátiles: desaparecen al reiniciar. Existen varias formas de hacerlas persistentes:

iptables-save y iptables-restore

bash# Exportar reglas a archivo (incluye todas las tablas)
iptables-save > /etc/iptables/rules.v4

# Restaurar desde archivo
iptables-restore < /etc/iptables/rules.v4

# Ver el formato del archivo exportado
cat /etc/iptables/rules.v4

iptables-persistent / netfilter-persistent (Debian/Ubuntu)

bashapt install iptables-persistent

# Guarda las reglas actuales en /etc/iptables/rules.v4
netfilter-persistent save

# Recarga las reglas guardadas
netfilter-persistent reload

Script en /etc/network/if-pre-up.d/

bash# Crear script que se ejecuta antes de levantar interfaces
cat > /etc/network/if-pre-up.d/iptables <<'EOF'
#!/bin/bash
iptables-restore < /etc/iptables/rules.v4
EOF
chmod +x /etc/network/if-pre-up.d/iptables
💡 En sistemas RHEL/CentOS con iptables-services, las reglas se guardan en /etc/sysconfig/iptables y se gestionan con service iptables save.

07 Consejos de Seguridad y Buenas Prácticas

Whitelist SSH ANTES de cambiar la política a DROP. Siempre añade la regla de aceptación SSH antes de ejecutar iptables -P INPUT DROP. De lo contrario te quedarás sin acceso al servidor.

Orden de reglas: específicas primero, generales después. Las reglas más concretas (una IP, un puerto) deben aparecer antes que las genéricas. El motor las evalúa en orden y para en la primera coincidencia.

Vacía las reglas antes de aplicar un script nuevo. Inicia siempre con iptables -F y iptables -X para evitar reglas duplicadas o conflictivas de ejecuciones anteriores.

Usa -n en los listados. Evita la resolución DNS inversa con iptables -L -n -v, lo que hace los listados mucho más rápidos en servidores con DNS lento.

Añade reglas con -I (insert) en lugar de -A (append) para máxima prioridad. Las reglas de bloqueo urgentes deben ir al inicio de la cadena.

Usa estados de conexión (conntrack). La regla -m state --state ESTABLISHED,RELATED -j ACCEPT es fundamental: permite que el tráfico de retorno fluya sin necesidad de reglas bidireccionales para cada servicio.

Prueba con un cron de seguridad. Antes de aplicar reglas arriesgadas en remoto, programa un cron que resetee las reglas en 5 minutos: echo "iptables -F" | at now + 5 min. Cancélalo si todo funciona.

Documenta tus reglas. Usa comentarios en los scripts y añade --log-prefix descriptivos en los LOG para facilitar el diagnóstico posterior.

Plantilla de script de firewall recomendada

bash#!/bin/bash
# firewall.sh — Plantilla de firewall segura

set -e   # Salir si hay error

# ══ 1. Limpiar reglas existentes ══
iptables -F
iptables -X
iptables -t nat -F
iptables -t mangle -F

# ══ 2. Políticas por defecto (PERMISIVAS al inicio) ══
iptables -P INPUT   ACCEPT
iptables -P FORWARD ACCEPT
iptables -P OUTPUT  ACCEPT

# ══ 3. Reglas específicas ══
iptables -A INPUT -i lo -j ACCEPT
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
iptables -A INPUT -p tcp --dport 22 -j ACCEPT   # SSH
# ... añadir más reglas aquí ...

# ══ 4. Políticas restrictivas AL FINAL ══
iptables -P INPUT   DROP
iptables -P FORWARD DROP

# ══ 5. Guardar ══
iptables-save > /etc/iptables/rules.v4
echo "Firewall aplicado correctamente."

08 Posibles Problemas y Soluciones

❌ Me quedé sin acceso SSH al servidor
▸ Solución

Si tienes acceso a la consola física o panel VNC/KVM del proveedor:

# Desde consola local: limpiar todas las reglas (acceso inmediato)
iptables -F
iptables -P INPUT ACCEPT

Si usas un proveedor cloud, utiliza la consola de emergencia (EC2 Serial Console, Azure Bastion, etc.).

❌ Las reglas desaparecen tras reiniciar
▸ Solución

Las reglas de iptables son volátiles por defecto. Usa alguno de los métodos de la sección de persistencia: iptables-persistent en Debian/Ubuntu, service iptables save en RHEL, o un servicio systemd personalizado.

❌ El NAT funciona pero los equipos internos no salen a Internet
▸ Solución 
# Verificar que IP forwarding está activo
cat /proc/sys/net/ipv4/ip_forward
# Debe devolver 1; si es 0:
echo 1 > /proc/sys/net/ipv4/ip_forward

# Verificar que FORWARD no está en DROP
iptables -L FORWARD -n

# Verificar la regla MASQUERADE
iptables -t nat -L POSTROUTING -v -n
❌ El DNAT (port forwarding) no funciona desde el propio gateway
▸ Solución

PREROUTING no se aplica al tráfico originado en el mismo equipo. Añade una regla en la cadena OUTPUT de la tabla nat:

iptables -t nat -A OUTPUT \
         -p tcp --dport 2222 \
         -j DNAT --to-destination 10.0.0.5:22
❌ Error: "iptables: No chain/target/match by that name"
▸ Solución

El módulo del kernel requerido no está cargado. Carga los módulos necesarios:

# Para el módulo 'recent' (rate limiting)
modprobe xt_recent

# Para el módulo 'state' / conntrack
modprobe ip_conntrack

# Ver módulos cargados relacionados con netfilter
lsmod | grep xt_
❌ Conflicto con firewalld o ufw
▸ Solución

iptables, firewalld y ufw gestionan netfilter por debajo. Usarlos simultáneamente produce conflictos. Elige uno:

# Deshabilitar firewalld (RHEL/CentOS)
systemctl stop firewalld
systemctl disable firewalld

# Deshabilitar ufw (Debian/Ubuntu)
ufw disable
⚠️ Entornos de prueba: practica siempre en máquinas virtuales (VirtualBox, KVM, VMware) o contenedores Docker/LXC antes de aplicar estas configuraciones en producción. Un error en un servidor de producción remoto puede dejarlo inaccesible.
