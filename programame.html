Plantilla de equipo: Interface + Solver + Main + Tests

En vez de “todo en Main”, usamos una interfaz para desacoplar la lógica del problema. El equipo cambia de problema creando otro Solver, sin tocar la infraestructura.

Entrada→ Case→ ProblemSolver→ Salida = menos WA, más velocidad y mejor reparto de tareas.

Incluye

  • Script Bash para crear proyecto Maven
  • Interfaz ProblemSolver + record Case
  • Ejemplo completo: “Mafalda cucharadas”
  • Plantilla Main con FastScanner
  • JUnit 5 con ejemplos y bordes
  • Patrón de resolución + roles de alumnos

1) Script de arranque (añadir al inicio)

Bash + Maven Archetype

Guarda esto como crear_proyecto.sh, dale permisos y ejecútalo.

crear_proyecto.sh 
#!/bin/bash
# Da permisos de ejecución 'chmod+x' primero 
mvn archetype:generate -DarchetypeGroupId=org.apache.maven.archetypes -DarchetypeArtifactId=maven-archetype-quickstart -DarchetypeVersion=1.5 -DgroupId=es.iescamas.programame -DartifactId=ProgramaMe -Dversion=1.0.0-SNAPSHOT -DjavaCompilerVersion=17  -DjunitVersion=5.11.0  -DinteractiveMode=false

Comandos útiles

  • chmod +x crear_proyecto.sh
  • ./crear_proyecto.sh
  • cd ProgramaMe
  • mvn test (ejecuta JUnit)
  • mvn -q -DskipTests package (compilar rápido)

Regla de entrega (importante)

  • En el juez online suele bastar con Main (sin paquetes).
  • Para entrenar en clase, mantener paquetes y tests es ideal.
  • Antes de enviar, puedes copiar-pegar el contenido de Main en el juez.

2) Enfoque con Interfaces (plantilla profesional)

Contrato + Caso + Solver

Objetivo: el equipo trabaja contra un contrato. Si cambia el problema, solo cambia el Solver (y su Case), sin romper el resto.

src/main/java/es/iescamas/programame/ProblemSolver.java 
package es.iescamas.programame;

/**
 * Contrato genérico para concurso:
 * - Recibe un caso (I)
 * - Devuelve una salida (O)
 */
public interface ProblemSolver<I, O> {
    O solve(I input);
}
src/main/java/es/iescamas/programame/MafaldaCase.java 
package es.iescamas.programame;

/**
 * Un "caso" del problema: a b n
 */
public record MafaldaCase(int a, int b, int n) { }
src/main/java/es/iescamas/programame/MafaldaSpoonsSolver.java 
package es.iescamas.programame;

/**
 * Implementación del problema (cucharadas).
 *
 * Interpretación:
 * - Mafalda: comió x >= a
 * - Padre: dejó n-x <= b  => x >= n-b
 * - 0 <= x <= n
 *
 * Respuestas posibles: x en [ max(a, n-b), n ]
 * Conteo: n - L + 1, si L <= n; si no, 0.
 */
public class MafaldaSpoonsSolver implements ProblemSolver<MafaldaCase, Integer> {

    @Override
    public Integer solve(MafaldaCase c) {
        int L = Math.max(c.a(), c.n() - c.b());
        if (L > c.n()) return 0;
        return c.n() - L + 1;
    }
}

Ventajas para el equipo

  • Analista define el Case y escribe tests.
  • Programador implementa el Solver.
  • Tester mete bordes y valida formato / tipos.

3) Main estándar ProgramaMe (solo I/O)

Main delega en Solver

Main se limita a leer casos y a imprimir resultados. La lógica vive en el solver (testeable).

src/main/java/es/iescamas/programame/Main.java 
package es.iescamas.programame;

import java.io.*;

public class Main {

    public static void main(String[] args) throws Exception {
        FastScanner fs = new FastScanner(System.in);
        StringBuilder out = new StringBuilder();

        ProblemSolver<MafaldaCase, Integer> solver = new MafaldaSpoonsSolver();

        int t = fs.nextInt();
        for (int i = 0; i < t; i++) {
            int a = fs.nextInt();
            int b = fs.nextInt();
            int n = fs.nextInt();

            int ans = solver.solve(new MafaldaCase(a, b, n));
            out.append(ans).append('\n');
        }

        System.out.print(out);
    }

    // Scanner rápido para concursos
    static class FastScanner {
        private final InputStream in;
        private final byte[] buffer = new byte[1 << 16];
        private int ptr = 0, len = 0;

        FastScanner(InputStream in) { this.in = in; }

        private int read() throws IOException {
            if (ptr >= len) {
                len = in.read(buffer);
                ptr = 0;
                if (len <= 0) return -1;
            }
            return buffer[ptr++];
        }

        int nextInt() throws IOException {
            int c;
            do { c = read(); } while (c <= ' ');

            int sign = 1;
            if (c == '-') { sign = -1; c = read(); }

            int val = 0;
            while (c > ' ') {
                val = val * 10 + (c - '0');
                c = read();
            }
            return val * sign;
        }
    }
}

Nota de concurso (juez online)

Si el juez exige una única clase Main sin paquetes, podéis “aplanar”: copiar el contenido del solver dentro de Main. Para entrenar en equipo, mantener interfaz + tests es mejor.

4) Pruebas unitarias (JUnit 5) — usando la interfaz

TDD recomendado en entreno

Las pruebas se hacen contra ProblemSolver. Así el equipo puede refactorizar la implementación sin cambiar los tests.

src/test/java/es/iescamas/programame/MafaldaSpoonsSolverTest.java 
package es.iescamas.programame;

import static org.junit.jupiter.api.Assertions.*;
import org.junit.jupiter.api.Test;

class MafaldaSpoonsSolverTest {

    private final ProblemSolver<MafaldaCase, Integer> solver = new MafaldaSpoonsSolver();

    @Test
    void ejemplos() {
        // 2 2 5 -> comidas posibles: 3,4,5 => 3
        assertEquals(3, solver.solve(new MafaldaCase(2, 2, 5)));

        // 7 5 10 -> comidas posibles: 7,8,9,10 => 4
        assertEquals(4, solver.solve(new MafaldaCase(7, 5, 10)));
    }

    @Test
    void bordes() {
        // Debe comer todo
        assertEquals(1, solver.solve(new MafaldaCase(5, 0, 5)));

        // Padre permite mucho: manda 'a' -> 2..5
        assertEquals(4, solver.solve(new MafaldaCase(2, 999, 5)));

        // Imposible: no puede comer más que n
        assertEquals(0, solver.solve(new MafaldaCase(6, 0, 5)));
    }
}

6) Patrón de resolución (el que más funciona en ProgramaMe)

Lectura → Modelo → Bordes → Código → Envío

Pipeline fijo (siempre igual)

  1. Leer Entrada/Salida y restricciones (primero).
  2. Modelar: variables + significado (crear record Case).
  3. Convertir historia a reglas (desigualdades / condiciones).
  4. Bordes: mínimo, máximo, “imposible”, “todo vale”.
  5. Tests: 2 ejemplos + 2 bordes (entreno TDD).
  6. Implementar Solver.
  7. Main solo I/O + formateo de salida (1 línea por caso).

Checklist antes de enviar

  • ¿Una línea por caso?
  • ¿Sin textos extra?
  • ¿Lee todos los casos?
  • ¿Tipos correctos (int/long)?
  • ¿Pasa ejemplo del enunciado?

7) Roles del equipo (3 alumnos) y cómo encaja con interfaces

Rotables por problema

🧠 Analista

  • Define el record Case (modelo del caso).
  • Traduce historia → reglas claras.
  • Escribe tests: ejemplo + bordes.
  • Valida el enfoque con el equipo.

💻 Programador

  • Implementa el Solver (interface).
  • Evita tocar Main (salvo lectura del caso).
  • Cuida complejidad y tipos.
  • Refactor mínimo y seguro.

🔍 Tester / Validador

  • Probar manualmente con ejemplos.
  • Busca “off-by-one”, límites y formato.
  • Propone casos trampa.
  • Da el “OK para enviar”.

Dinámica recomendada

Rotad roles cada 1–2 problemas. El enfoque con interfaces ayuda a repartir tareas: mientras el programador implementa, el analista puede preparar tests del siguiente problema.
