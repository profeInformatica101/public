Rúbrica DAM · Proyecto Final

Evaluación integral del proyecto: análisis, diseño, desarrollo multiplataforma, UX, calidad, documentación, pruebas y defensa.

FP Grado Superior DAM Total 100%

Cómo evaluar

  • Selecciona el nivel por criterio y justifica con evidencias: commits, issues/PR, capturas, memoria, tests, demo.
  • Recomendado: repositorio Git + guía de instalación + memoria técnica estilo TFG + presentación.
  • La nota final se calcula por ponderación de bloques (tabla al final).

Bloque 1 · Análisis, Planificación y Diseño (20%)
Arquitectura · Requisitos · Modelado
1.1 Análisis de requisitos
Funcionales + no funcionales · trazabilidad
 Requisitos completos, claros y verificables. Incluye NFR (seguridad, rendimiento, usabilidad) y matriz de trazabilidad (requisito ↔ casos de uso ↔ implementación ↔ pruebas).
Razón: control total del alcance y validación con evidencias.
 Requisitos sólidos y estructurados. NFR presentes con menor detalle o trazabilidad parcial.
Razón: buen análisis, mejora posible en cobertura y evidencias.
 Requisitos correctos y entendibles. NFR poco detallados y trazabilidad limitada.
Razón: define el “qué”, pero el “cómo se valida” queda débil.
 Requisitos básicos y genéricos. Ambigüedades o falta de criterios de aceptación.
Razón: existe alcance, pero aumenta riesgo de desviaciones.
 Requisitos incompletos, confusos o inexistentes.
Razón: impide medir cumplimiento de forma objetiva.
1.2 Diseño técnico y arquitectura
Capas · componentes · patrones · diagramas
 Arquitectura coherente y escalable (capas/clean). Diagramas completos (casos de uso, clases, secuencia, despliegue). Patrones aplicados y justificados.
Razón: maximiza mantenibilidad y defensa técnica.
 Diseño consistente y correcto. Diagramas principales presentes, con algún detalle ausente.
Razón: sólido para entender y evolucionar, con mejoras de completitud.
 Diseño funcional con decisiones poco justificadas o diagramas parciales.
Razón: suficiente para construir, pero resta claridad técnica.
 Diseño mínimo. Arquitectura poco definida o diagramas muy básicos.
Razón: eleva deuda técnica y riesgo de integración.
 Sin diseño válido o con incoherencias graves.
Razón: proyecto frágil y difícil de evaluar/defender.
Bloque 2 · Desarrollo Multiplataforma (30%)
Programación · Servicios · Acceso a datos
2.1 Calidad del código
POO · SOLID · legibilidad · errores
 Código limpio y mantenible: modular, sin duplicidad, naming consistente, manejo robusto de errores, logging, configuración externa y estilo homogéneo.
Razón: calidad profesional y mantenibilidad alta.
 Código bien organizado y legible. Refactors menores o documentación parcial pendientes.
Razón: nivel profesional correcto con margen de pulido.
 Código funcional con repetición o estructura mejorable.
Razón: cumple, pero incrementa deuda técnica.
 Código básico y poco consistente. Patrones o estructura rígida.
Razón: difícil de ampliar y mantener.
 Código incompleto, fallos graves o no funcional.
Razón: no garantiza el logro del proyecto.
2.2 Servicios, procesos y concurrencia
Tareas · resiliencia · comunicación
 Implementación robusta: gestión de concurrencia sin bloqueos, cancelación, reintentos, timeouts y tolerancia a fallos. Comunicación clara y segura.
Razón: comportamiento fiable en escenarios reales.
 Servicios/procesos correctos con concurrencia adecuada. Falta alguna mejora de resiliencia.
Razón: estable con margen de robustez.
 Funcional, pero con manejo limitado de errores o concurrencia.
Razón: puede fallar en condiciones no ideales.
 Uso mínimo de servicios/procesos y solución poco escalable.
Razón: evidencia básica con fragilidad.
 No implementa lo requerido o presenta fallos críticos.
Razón: no evidencia competencia del bloque.
2.3 Persistencia y acceso a datos
DAO/Repository · transacciones · rendimiento
 Persistencia eficiente y segura: transacciones correctas, validación, consultas optimizadas, índices, migraciones/versionado de BD y manejo de errores.
Razón: consistencia, rendimiento y mantenibilidad asegurados.
 Persistencia correcta y estructurada. Alguna optimización pendiente o documentación parcial.
Razón: buen nivel con margen de mejora.
 Persistencia funcional con consultas poco optimizadas o estructura mejorable.
Razón: cumple, pero puede degradar rendimiento y claridad.
 Persistencia básica y limitada. Riesgos de integridad o fallos en casos límite.
Razón: evidencia mínima, solución frágil.
 Persistencia inexistente o incorrecta.
Razón: no cumple un pilar esencial del proyecto.
Bloque 3 · Interfaces y UX (15%)
UI · Accesibilidad · Multimedia
3.1 Interfaz y usabilidad
Consistencia · accesibilidad · feedback
 UI intuitiva y consistente, accesible (contraste, foco, teclado), con feedback claro (carga/errores/éxito), y adaptada a plataforma/pantalla. Guía de estilos.
Razón: experiencia de usuario profesional.
 UI clara y funcional. Accesibilidad correcta con detalles mejorables.
Razón: buena UX con ajustes menores pendientes.
 UI correcta pero básica. Inconsistencias visuales o de interacción.
Razón: cumple sin pulido de producto final.
 UI mínima o poco cuidada. Dificulta tareas en algunos flujos.
Razón: usable, pero con fricción notable.
 UI confusa, rota o inexistente.
Razón: impide validar el proyecto correctamente.
3.2 Gestión multimedia / recursos
Archivos · permisos · rendimiento
 Gestión correcta de recursos multimedia con optimización (tamaño/caché), permisos, formatos y manejo de errores.
Razón: integración eficiente y realista.
 Integración correcta, con optimización parcial o documentación mejorable.
Razón: funciona bien, falta pulir casos límite.
 Integración básica sin optimización.
Razón: cumple, pero puede penalizar recursos o UX.
 Uso mínimo o poco estable.
Razón: evidencia limitada y frágil.
 No implementa o falla de forma crítica.
Razón: no cumple objetivos del bloque.
Bloque 4 · Entornos, Git y Despliegue (10%)
DevTools · CI/CD · reproducibilidad
4.1 Git y flujo de trabajo
Commits · ramas · PR · issues
 Flujo profesional: commits semánticos, ramas por feature, PR con revisión, issues bien descritas, tags/releases y changelog.
Razón: máxima trazabilidad y trabajo “real”.
 Buen uso de Git con ramas y commits adecuados. Falta alguna práctica (releases/PR más completos).
Razón: control correcto con formalización mejorable.
 Uso básico: commits y alguna rama, sin metodología consistente.
Razón: trazabilidad limitada del proceso.
 Historial poco útil o commits esporádicos.
Razón: difícil evaluar evolución y aportación.
 Sin control de versiones utilizable.
Razón: sin evidencias fiables del desarrollo.
4.2 Build y despliegue
Reproducibilidad · scripts · instalación
 Build reproducible, scripts de instalación/ejecución, configuración por entorno y despliegue funcional con guía clara.
Razón: “plug & play” para evaluación y uso real.
 Build y ejecución correctos. Documentación buena, automatización mejorable.
Razón: funciona bien, con algún paso manual extra.
 Ejecuta con varios pasos manuales o dependencias poco explicadas.
Razón: aumenta el riesgo de fallo en evaluación.
 Dificultad para ejecutar; guía incompleta.
Razón: reproducibilidad limitada.
 No compila/ejecuta o no hay guía válida.
Razón: no evaluable de forma fiable.
Bloque 5 · Documentación Profesional (TFG) (15%)
Memoria · Manuales · API
5.1 Memoria técnica estilo TFG
Estructura · rigor · evidencias
 Memoria completa y profesional: objetivos, análisis, diseño, implementación, pruebas, resultados, conclusiones, mejoras futuras, bibliografía y anexos (diagramas, capturas, tablas).
Razón: documento defendible y reutilizable.
 Memoria muy correcta y completa, con menor profundidad en anexos/justificación técnica.
Razón: sólida con margen para enriquecer evidencias.
 Memoria correcta pero más descriptiva que técnica. Faltan apartados o evidencias.
Razón: explica el proyecto, pero lo demuestra parcialmente.
 Memoria básica, incompleta o desorganizada.
Razón: difícil validar decisiones y resultados.
 Sin memoria válida o con calidad insuficiente.
Razón: no cumple el estándar de un TFG profesional.
5.2 Documentación del repositorio
README · instalación · manual usuario · API
 README excelente (arquitectura, instalación, ejecución, configuración, pruebas, troubleshooting). Incluye manual de usuario y documentación API si aplica.
Razón: reduce soporte y facilita evaluación.
 Documentación completa y clara. Falta algún detalle (troubleshooting/API formal).
Razón: adecuada y profesional, mejorable en estandarización.
 README correcto pero incompleto. Manual o API poco desarrollados.
Razón: ayuda, pero no evita dudas durante despliegue.
 Documentación mínima.
Razón: alto coste de soporte y riesgo de errores.
 Documentación inexistente o confusa.
Razón: no mantenible ni evaluable con garantías.
Bloque 6 · Pruebas y Defensa (10%)
QA · Demo · Comunicación
6.1 Pruebas (QA)
Unitarias · integración · evidencias
 Estrategia clara + tests relevantes (unit/integration), casos límite y evidencias (report/log/capturas).
Razón: demuestra fiabilidad y calidad.
 Buen conjunto de pruebas. Falta ampliar casos límite o automatización.
Razón: valida bien con margen de mejora.
 Pruebas básicas centradas en flujos principales.
Razón: valida lo esencial, riesgo en escenarios reales.
 Pocas pruebas o principalmente manuales.
Razón: calidad poco demostrada.
 Sin pruebas o fallos críticos sin control.
Razón: no se garantiza funcionamiento.
6.2 Defensa y demo
Estructura · rigor técnico · preguntas
 Presentación clara y técnica + demo fluida. Responde con solvencia y justifica decisiones (arquitectura, datos, UX, seguridad, pruebas).
Razón: evidencia dominio y criterio profesional.
 Defensa sólida y demo correcta. Alguna imprecisión menor.
Razón: buen dominio con pequeños vacíos.
 Defensa correcta pero más descriptiva que técnica. Demo con pequeños problemas.
Razón: justificación técnica mejorable.
 Defensa básica y demo limitada. Dificultades en preguntas.
Razón: dominio parcial del proyecto.
 No defiende adecuadamente o demuestra desconocimiento.
Razón: invalida la autoría y el aprendizaje alcanzado.

Checklist mínimo para aprobar

  • Repositorio con README funcional y guía de instalación reproducible.
  • Aplicación ejecutable y demostrable (demo estable).
  • Persistencia y acceso a datos funcionando correctamente.
  • Interfaz usable (sin bloqueos de uso).
  • Memoria técnica con estructura mínima tipo TFG.
  • Alguna evidencia de pruebas (manuales o automatizadas).
